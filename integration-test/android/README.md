# Debuggable — Android Sample

A minimal Android app that applies the `me.tbsten.debuggablecompilerplugin` plugin to a `ViewModel` implementing `AutoCloseable`.

## Prerequisites

- Android SDK installed (the build reads `sdk.dir` from `local.properties`)
- Publish the plugin and runtime to your local Maven repository from the repo root:

```bash
cd ../../           # back to repo root
./gradlew publishToMavenLocal
```

Re-run this whenever you change the plugin or runtime.

## Build

```bash
./gradlew :app:assembleDebug
```

The debug APK lands at `app/build/outputs/apk/debug/app-debug.apk`.

## Run on a device/emulator

```bash
./gradlew :app:installDebug
adb logcat | grep Debuggable
```

Each click on `+1` or `Next label` prints a `[Debuggable]` log line to Logcat:

```
[Debuggable] count: 1
[Debuggable] label: Loading
[Debuggable] action: increment()
```

### Excerpt — `MainActivity.kt`

```kotlin
private val STATUSES = listOf("Idle", "Loading", "Running", "Success", "Error", "Done")

@Debuggable
class CounterViewModel : ViewModel(), AutoCloseable {
    private val _count = MutableStateFlow(0)
    val count: StateFlow<Int> = _count.asStateFlow()    // ← auto-tracked

    private val _label = MutableStateFlow(STATUSES.first())
    val label: StateFlow<String> = _label.asStateFlow() // ← auto-tracked

    private var labelIndex = 0

    fun increment() { _count.update { it + 1 } }        // ← logs on call
    fun nextLabel() {                                   // ← logs on call
        labelIndex = (labelIndex + 1) % STATUSES.size
        _label.value = STATUSES[labelIndex]
    }

    override fun close() {
        // Plugin wraps this in try-finally so the registry is released
        // when the ViewModel is cleared.
    }
}
```

## Verify the plugin ran

After compiling, inspect the generated bytecode:

```bash
javap -p -c app/build/tmp/kotlin-classes/debug/example/debuggable/android/CounterViewModel.class \
  | grep -E "debuggable|logAction"
```

You should see:
- A private field `$$debuggable_registry` of type `DebugCleanupRegistry`
- Calls to `debuggableFlow(...)` and `logAction(...)`

## AutoCloseable behaviour

Because `CounterViewModel` implements `AutoCloseable`, the plugin:
1. Injects a `$$debuggable_registry` field initialised per-instance
2. Routes each `debuggableFlow(...)` call through that registry (not the shared default)
3. Wraps `close()` in a `try { original body } finally { registry.close() }` so all Flow observations are cancelled when the ViewModel is cleared

This is the recommended pattern for anything with a lifecycle.
